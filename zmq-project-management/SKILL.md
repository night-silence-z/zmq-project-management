# 项目管理规范（zmq-project-management）

- 版本：v1.2（2026-07-17；修订历史见同目录 `CHANGELOG.md`，不随会话加载）
- 设计原则：结构化、记录、检查、收口由 AI 承担，不要求使用者手工填写模板；所有阈值与命名风格可按团队习惯裁剪。

## 1. 新项目启动：立项访谈（一次性，收敛即止）

问清三要素：最终产出物是什么形态、给谁用（+拿去干什么）、如何验收（可观察标准）。

- 逐轮追问直到收敛，每轮 ≤3 问；能给选项就给"选项+推荐"供挑选或否决，不出开放式长问卷；用户答"你来定"则 AI 定并说明理由。
- 继续追问的条件：三要素任一模糊，或目标含未定义的抽象词（"优化""升级"类）。
- 用户说"先干起来"：立即停止追问，已知写入启动卡，缺口标"待确认"转入执行。
- 收敛判据：能写出"本项目交付 X，给 Y 用来 Z，验收标准 W"+范围三栏（做/不做/顺延）。
- 产出两件：① 一页启动卡按 `references/启动卡模板.md` 写入 `00_管理/README.md`；② 在项目根 `CLAUDE.md` 写一行"本项目按 zmq-project-management skill 管理，会话开工/收工按其例程执行"——让项目自己拉起本 skill，不依赖触发词命中（使用其他 AI 工具时写入对应入口文件，如 `AGENTS.md`）。
- 用户签核后访谈结束；执行期不再重复问（例外：台账待确认项到达截止条件时，按约定回到用户处要决策）。
- 规划期与实现期分开：用户说"先不写代码"就不碰实现。

## 2. 存量项目接管（已有项目结构混乱/不合规时）

1. 只读盘点，不动任何文件。
2. 产出现状清单：现有目录、各产出物的疑似当前版（多个并存全部列出）、台账与口径缺口、旧结构→新结构映射表。
3. 疑似当前版冲突交用户裁决：列候选+推荐，一次问清。
4. 批量移动/重组文件影响共享现场：方案经用户确认后执行，结果在台账登记。
5. 补建 README（含指针表）与台账：从现有材料回填关键决策与版本关系，不虚构历史，缺失处标"待查"；此后进入正常例程，并写入项目自举行（同第 1 节）。

## 3. 目录与项目根

- 每个项目一个独立根目录（目录名即项目名），互不嵌套；会话开在父目录/工作区时，先确认目标项目再进入其根目录工作。
- 基础 4 目录，用到才建，空目录不建：

```
00_管理/
  README.md      总览 + 当前状态 + 当前版指针表 + 权威源清单
  台账.md        三节：待办 / 决策与待确认 / 迭代记录
01_产出/
  当前/  历史/   （单一产出物时的扁平结构）
02_输入/         原料、参照物、外部文档
99_归档/         废稿（未进入版本线或整线被放弃的产物）、切片台账、一次性文件
```

- 按需扩展：分析/汇报类在 `00_管理/` 加 `口径字典.md` 或 `术语与红线.md`（见第 6 节）；数据分析加 `03_数据/`、`04_脚本/`；开发类代码走 git 仓库、资料区加 `03_验收记录/`、`00_管理/` 加 `契约清单.md`；多期项目在 `01_产出/` 内按期分 `10_一期/ 20_二期/`——期在外、线在内。
- **建档用脚手架**：运行 `python scripts/init_project.py 项目名 [--type 分析|开发|汇报]`——一键建基础目录、README/台账骨架和自举行；随后按 `references/README模板.md`、`references/台账模板.md`、`references/启动卡模板.md` 填内容（模板只在建档时读取，内含填好示例，照示例颗粒度写）。无法运行脚本时手工按模板建。
- 交接文档不常驻：仅在用户提出交接、或项目收尾/长期挂起时现场汇编生成（内容重叠的常驻文件必漂移）。

## 4. 产出物与版本管理

- **铁律**：交付物（用户要肉眼看、传阅的东西——HTML/报告/原型/PRD）新增文件 `名称-vN.扩展名`，绝不覆盖；每升一个 vN 在台账迭代记录登记一行（变更+被否决方案）。
- **旧版去向判据**：被新版本取代 → 移入所在线的 `历史/`（版本血缘不拆散，支持"回归 v5"式指名回滚）；整线或整份被放弃 → 移入 `99_归档/`，整线连同其内部 `当前/历史` 整体搬移、不拆散。`99_归档/` 不收正常旧版；移动是"搬"不是"拷"，同一文件任一时刻只在一处。
- **多类型产出物按"产出物线"拆分**（一条线 = 独立演进、独立版本号、独立验收的一类交付物）：

```
01_产出/
  PRD/    当前/ 历史/
  原型/   当前/ 历史/
  汇报/   当前/ 历史/
```

- 第二种类型的产出物出现时才拆线；单一产出物保持扁平结构。
- **脚本与代码的归属**：有 git 仓库的一律走 git（改动即 commit，不用 -vN）；无 git 的轻量项目，脚本放 `04_脚本/`、以 -vN 演进、旧版进 `99_归档/`（这是归档收"正常旧版"的唯一例外——脚本平铺无 `历史/` 可去）；脚本本身是交付物（要外发给他人使用）时，才按产出物线管理。
- README 维护**当前版指针表**：`产出物线 | 当前版文件 | 状态（用第 7 节阶梯词）| 对应关系`；跨线映射（如原型 ↔ PRD vN）必须登记，防线间版本漂移。
- **git 低开销例程**（git 不拖慢会话，大输出进上下文才拖慢）：对账只跑 `git log --oneline -5` + `git status -s`；看变更用单文件 diff 或 `--stat`，不把全仓 diff 贴进对话；稳定节点才打标签；本地 commit 自主，push/合并/部署逐次问。

## 5. 台账（轻量，AI 维护）

- 默认单文件 `00_管理/台账.md` 三节：待办 / 决策与待确认 / 迭代记录。
- 迭代记录行：`时间(到分钟) | 阶段 | 事件类型{进展/决策/重构/推翻/纠错/阻塞} | 内容 | 原因 | 影响与关联文档`；推翻和纠错写清"原来是什么 → 改成什么 → 为什么"。
- 决策编号 D-xxx；待确认项写成"事项 | 决策人 | 截止条件 | 默认方案 | 影响范围"，目标是消灭悬置项。
- 只记关键决策、边界、版本、待确认，不记闲聊；用户要求记账时立即落账。
- 记录须具体可追溯：❌"修了一些问题、优化了体验"；✅"纠错：楼盘卡字段原按 v0.2 契约 → 改按 v0.4 实时聚合，因后端接口文档更新"。
- 拆分触发（任一满足，AI 主动提议）：迭代记录超 50 条 / 台账超 300 行 / 多 AI 或多人并行写入 → `迭代总账.md` 独立只增，`台账.md` 留作入口（待办+决策+最近 10 条迭代摘要+指向）；总账按阶段切片归档进 `99_归档/`，主文件只留活跃段。
- 多 AI 协作时共用一套台账，各自落账时在"内容"列注明执行者，不各建一套。
- 表格行间不留空行（条件规则：部分平台如飞书同步时空行会使表格断裂；无此场景可忽略）。

## 6. 口径唯一性（分析/汇报类重点）

1. **唯一权威源**：分析类 = `口径字典.md`，开发类 = `契约清单.md`，汇报类 = `术语与红线.md`，放 `00_管理/`；README 登记"哪个文件管哪类口径"。
2. **引用不复制**：任何定义只在权威源出现一次，产出只引用（"口径见字典 vN"），不得自带定义；例外：由机器从权威源生成、并标注字典版本的口径附录。改口径先改权威源并登记变更行（旧→新→原因），再全链路同步产出。
3. **交付前机器扫描**：从权威源提取术语/口径/版本号清单，对产出全文做冲突扫描——可运行本 skill `scripts/check_terms.py`（读取权威源中"禁用词|应使用"表，扫描产出目录并报告命中行，默认跳过 历史/ 与 99_归档/）；语义级不一致扫描抓不全，由第 9 节整体扫描兜底。

## 7. 阶段、范围与交付状态阶梯

- 中大型项目按阶段闸门推进（M0…Mn），每阶段有目标、范围、验收标准、可回退点。
- 范围可砍（顺延/不做清单），已定部分的事实、口径、语义零容忍打折。
- 探索性产出标注"探索件，不代表已决策"；未纳入本期的既有能力标注"保留"，防误删。
- **交付状态阶梯**（指针表与汇报只用阶梯词——❌"差不多好了""基本完成"；✅"候选——待用户验收"）：文档/设计线：探索 → 候选 → 已定稿 → 已交付；开发线：已定义 → 本地通过 → 测试环境通过 → 已上线验收。

## 8. 会话例程（补偿 AI 无跨会话记忆，每次会话执行）

**开工清单**（复制进回复，逐项勾选）：

```
- [ ] 读 README：总览 + 当前状态 + 当前版指针表
- [ ] 读台账：待办、决策与待确认
- [ ] 读迭代记录最近 20 条
- [ ] （代码项目）git log --oneline -5 + git status -s
- [ ] 台账与文件系统有冲突？→ 以文件系统为事实，先在台账落一行纠错（原记录→实际情况→原因，不明标"待查"）再干活
```

**收工清单**：

```
- [ ] 落账本次进展/决策/推翻（推翻写清 原来→改成→为什么）
- [ ] 更新待办与当前版指针表
- [ ] 列出待用户授权的同步/推送事项
```

- 没落进台账的内容等于没发生——多 AI 协作时，台账是彼此了解对方做了什么的唯一通道。

## 9. 一致性收口（AI 主动，可观察触发）

- 任何事实/口径/版本变更后，同步检查三处：README（含指针表）、权威源、台账。
- **整体扫描触发**（满足其一即扫，不依赖"轮数"感觉）：① 迭代记录自上次扫描新增满 20 条；② 任一产出物线状态升为"已定稿"；③ 项目收尾。扫描完成后在台账登记"一致性扫描已至第 X 条（日期）"。
- 机械部分下沉给脚本：运行 `python scripts/health_check.py 项目根`——自动核查指针表指向的文件是否存在、状态是否为阶梯词、各线"当前/"是否唯一、水位是否滞后 ≥20 条、表格是否有空行。

**交付前清单**：

```
- [ ] health_check.py 零错误
- [ ] check_terms.py 零命中（有权威源禁用表时）
- [ ] 全文口径术语一致（语义级人工过一遍）
- [ ] 渲染/运行自检通过（裁切/折行/字号/跑版）
- [ ] 验证边界已声明（本地/测试/生产；自动测试/真人体验）
```

## 10. 授权边界

- 本地操作（写文件、本地 commit、装环境）自主执行。
- 不可逆或触达他人（push、部署、发布、外发、删除共享资产、同步到协作平台）默认停，逐次明确授权，一次授权不延伸到下次。
- 会误导执行的废弃内部材料：经授权删除，并在台账留一行"删了什么、为什么"。

## 11. 交接与收尾

- 交接文档按需生成（用户提出交接 / 收尾或长期挂起时）：主线、产物清单、约束、待办、"给新对话的快速上下文"——从 README+台账现场汇编，保证最新。
- 收尾五件事：① **对照启动卡的验收标准逐条核对**，未达标项明示并请用户裁决（补做/降级/顺延）；② 唯一终版标记+版本合流；③ 资产沉淀归位——可复用模板/脚本/规范移入跨项目资产库（位置由使用者指定：首次收尾时询问一次，记入 README 后沿用）；④ 按本 skill 目录 `references/复盘模板.md` 生成《用户工作方式与项目资产复盘》（收尾才读该文件，不占日常上下文）；⑤ 一致性收尾扫描（第 9 节，含 health_check.py）。

## 12. 本 skill 的维护

- 本 skill 随其源仓库演进：修改在仓库中进行，递增 frontmatter `metadata.version` 与头部版本行，并在 `CHANGELOG.md` 追加一行。
- 本地定制（改阈值、删条件规则等）建议 fork 后自立版本线；升级时对照 CHANGELOG 合并，避免定制被覆盖。
- 使用者的全局规则/记忆载体只保留指向本 skill 的指针和少量高频铁律，不复制条文（防双源漂移）。
