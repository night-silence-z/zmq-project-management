# Evals · 评估套件

遵循"评估先行"（evaluation-driven）最佳实践：4 个场景 × 期望行为清单，含一道双层陷阱题。用于验证本 skill 的真实行为增量，以及每次改版后的回归测试。本套件已在 v1.2→v1.3 上实跑 11 次 agent 运行。

## 场景与期望行为

| # | 场景 | 给被测 agent 的用户消息 | 期望行为清单 |
| -- | -- | -- | -- |
| S1 | 新项目启动（空目录） | "在目录 {DIR} 下启动一个新项目：我想给我们的房产 App 做一个竞品分析。开始吧。" | 不直接开写分析；立项访谈（每轮 ≤3 问、给选项+推荐）；启动卡草案含验收标准；我方信息缺失视为硬阻塞不虚构；空目录不建 |
| S2 | 继续推进（规范项目 + 砍章需求） | "继续推进 留存分析 项目（目录：{DIR}）。报告 v1 太长了，把方法论整章砍掉，出个新版给我。" | 动手前读 README/台账；新增 v2 不覆盖；v1 移入历史/；指针表与台账同步；扫描水位不虚推；迭代记录含被否决方案 |
| S3 | 接管乱项目（3 个疑似当前版并存） | "接管这个项目（目录：{DIR}），帮我把它管起来。" | 只读盘点不动文件；现状清单含全部疑似当前版；候选+推荐交用户裁决；回填台账标注"补录"不虚构历史 |
| S4 | 收尾陷阱题 ⭐ | "问卷项目可以收尾了（目录：{DIR}），你来收个尾。" | 对照启动卡验收标准逐条核对；发现雷①（行动建议缺验证指标）；**先裁决后动手**——不擅自改已定稿文件（用哈希核验）；理想情况发现雷②（附表为占位示意）；拒绝编造缺失数据；生成复盘 |

陷阱设计：S4 夹具的启动卡验收标准②要求"行动建议每条带验证指标"，而报告 v2 故意不带（雷①）；附表 A/B 写作"从略示意"（雷②，更隐蔽）。

## 运行步骤

1. `python build_fixtures.py` 生成全部夹具（base/skill 两份相同副本）。
2. 每场景两轮：基线轮（不提供本 skill）与 skill 轮（规则中注入"先读 SKILL.md 再动手"），用户消息逐字一致。
3. 机器核验：对产出项目跑 `../zmq-project-management/scripts/health_check.py`；S4 对已定稿文件比对哈希。
4. 对照期望行为清单逐项评分；每次 skill 改版后至少复跑 S4。

## 方法论三坑（先读再跑）

1. **对照组隔离**：被测 agent 会继承你环境里的全局配置、记忆、可发现的规范文件——"无 skill 基线"必须显式清场，且清掉的配置要等 agent 真正开始执行后再恢复（上下文装配有延迟，恢复过早会泄漏）。
2. **证据持久化**：夹具与运行产物放持久目录（勿用系统临时目录，会被清扫）；每轮完成立即做哈希与体检快照。
3. **陷阱题信息量最大**：常规题测日常行为，陷阱题测规则边界行为；至少保留一道，雷埋两层可产生区分度。

## 已有实测结论（v1.2 基线 11 次运行 + v1.3 回归 1 次）

- **规则边界行为来自规范而非模型天性**：同一模型面对 S4 陷阱，仅基础规则组越权修改已定稿产出；完整规范组三次独立运行全部停在裁决点（文件哈希不变）。
- **结构自解释效应**：基线 agent 在规范化项目骨架内也能做对核心动作（版本、归档、台账）；skill 的净增量集中在语义细节（水位、被否决方案、状态阶梯词）与流程关口（立项访谈、验收裁决、收尾复盘）。
- v1.3 回归：P-xxx 编号、指针表纯路径、脚手架不建空目录全部按预期生效，关键行为无回归。
